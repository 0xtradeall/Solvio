You are an expert Full-Stack Frontend Developer specializing in React-based Progressive Web Apps (PWAs) for blockchain applications on Solana. Your task is to generate the complete, production-ready frontend codebase for the Solvio project—a mobile-first PWA serving as a Solana payment hub for real-world use cases like freelancer invoicing, merchant payments, and bill splitting among friends. This app targets users on mobile devices (optimized for 390px+ screen widths, installable on home screens) and emphasizes simplicity, no traditional logins (Phantom wallet connection acts as authentication), client-side only operations (no backend or database), and modularity for future Android offline integration via Pollinet SDK.

### Project Overview and Core Requirements
Solvio enables seamless SOL and USDC payments without fiat on-ramps or other tokens (MVP scope). Key principles:
- **Authentication & Storage**: One-time connection to Phantom wallet via @solana/wallet-adapter-react. No private keys stored or handled—Phantom manages signing. Store receipts in browser localStorage, keyed to the connected wallet's public address for persistence across sessions on the same device (no cross-device sync needed). Example key: `solvio_receipts_${walletAddress}` as a JSON array of transaction objects.
- **UI/UX Design**: Clean, fintech-inspired interface like Revolut or Venmo. Use a purple (#7C3AED) and teal (#0D9488) color scheme. Mobile-optimized: Touch-friendly buttons (min 44px height), sans-serif fonts (e.g., Inter or system-ui), ample whitespace. Bottom tab navigation for tabs: Request (home), Split, Receipts, Settings. Ensure PWA manifest for installability (icons, theme color: purple).
- **No Backend**: All logic client-side. Transactions broadcast directly via Solana RPC (use mainnet-beta). Handle errors gracefully: Validate wallet addresses pre-send (using @solana/web3.js PublicKey validation), show red highlights for invalid fields, partial success summaries for splits (no rollbacks on confirmed txs), and individual Retry buttons for failures.
- **Features Breakdown**:
  1. **Request Tab**: Form for amount (SOL or USDC toggle), optional note. Auto-fill receiver address from connected wallet. Generate shareable URL (e.g., current domain + `/pay?amount=${amount}&currency=${currency}&address=${address}&note=${note}`) and large QR code (via qrcode.react, 300x300px, centered). On payer side: Link opens app/PWA, auto-triggers Phantom for one-tap payment. Post-confirmation: Poll for tx signature, then generate/download PDF receipt via jsPDF (include: date, amount/currency, sender/receiver addresses, tx ID hyperlinked to `https://solscan.io/tx/${txId}`, note). Save to localStorage.
  2. **Split Tab**: Input total amount (SOL/USDC), description, dynamic list of participants (add/remove via + button; each with nickname input and wallet address input—validate on blur). Toggle: Equal split (auto-calculate per person) or custom amounts (sum must equal total). "Send All Payments" button: Locks inputs, sends parallel transactions (use Promise.allSettled), shows live per-person status badges (Pending: yellow spinner; Confirmed: green check; Failed: red X with Retry). On partial success: Summary modal (e.g., "3/5 confirmed"). Generate single PDF for whole split (list all tx details). Save all to localStorage.
  3. **Receipts Tab**: List view of stored transactions (filter by date/currency if connected). Each card: Amount, date, status, participants (for splits), Download PDF button (regenerate via jsPDF), WhatsApp share button (construct `whatsapp://send?text=Receipt: ${amount} ${currency} from ${sender} to ${receiver}. Tx: https://solscan.io/tx/${txId}` and open via window.open). Empty state: "Connect wallet to view receipts."
  4. **Settings Tab**: Wallet connect/disconnect button, clear localStorage option (with confirmation), app info (version 1.0 MVP), and toggle for testnet (disabled for MVP). Disconnect clears session data.
- **Error Handling Specifics**:
  - Invalid wallet address: Highlight field red, show tooltip "Enter a valid Solana address (e.g., starts with 1-9A-HJ-NP-Z)."
  - Tx failure: Mark "Failed" in red, enable Retry (resend only that tx).
  - Partial splits: "Partial success: X confirmed, Y failed. Confirmed txs cannot be undone."
  - Network issues: Fallback to retry with exponential backoff (e.g., 1s, 2s, 4s).
  - Lock during sends: Disable inputs/buttons until all resolve.
- **Modularity for Future Integration**: Encapsulate all Solana logic in a standalone `lib/transactions.js` file. Export functions like `requestPayment(amount, currency, receiverAddress, note, onStatusUpdate)`, `splitPayments(participants, totalAmount, currency, description, onStatusUpdate)`, `getTransactionStatus(txId)`. Use a callback pattern: `onStatusUpdate({ type: 'pending'|'confirmed'|'failed', data: {...}, error? })` to allow future interception (e.g., by Pollinet SDK in Android wrapper—replace RPC calls with offline relay). Avoid any hardcoded RPC in UI components; inject Connection via context.
- **Compliance & Security**: No personal data collected (GDPR-irrelevant). Wallet addresses are public; link txs to Solscan safely. Never prompt for seeds/phrases.
